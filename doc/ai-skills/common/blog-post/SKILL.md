# Blog Post Skill - 博客文章处理规范

## 适用范围

本 skill 适用于处理本项目中与博客文章相关的所有任务，包括：
- 创建新博客文章
- 编辑/修改现有博客文章
- 格式化文章内容
- 优化文章 SEO
- 管理文章标签和分类

## 执行流程

### 1. 任务分析与确认

- 首先明确用户需求：是创建新文章还是修改现有文章
- 若创建新文章，确认以下信息：
  - 文章标题（必填）
  - 发布日期（默认当前日期）
  - 文章标签（可选，建议至少1个）
  - 文章分类（可选）
  - 内容格式（Markdown）

### 2. 文件位置与命名规范

- 文章存储目录：`_posts/`，按年份分类
- 文件名格式：`YYYY-MM-DD-文章标题.md`
- 示例：`2023-10-15-如何高效阅读一本书.md`

### 3. 文章内容规范

#### 3.1 前置元数据（Front Matter）

所有文章必须包含标准的 YAML 前置元数据：

```yaml
---
layout:     post
title:      "文章标题"
subtitle:   "副标题（可选）"
date:       2023-10-15 12:00:00
author:     "作者名（默认：enilu）"
header-img: "img/post-bg-2015.jpg"  # 可选，默认使用网站配置
catalog:    true
tags:
    - 阅读
    - 效率
---
```

#### 3.2 正文格式

- 使用 Markdown 格式编写
- 支持 GitHub Flavored Markdown (GFM)
- 代码块使用三个反引号 ``` 并指定语言
- 图片存储在 `img/` 目录，使用相对路径
- 引用使用 > 符号
- 标题层级使用 # 符号（最多三级）

#### 3.3 读书笔记类文章规范（核心重点）

对于读书笔记类文章，除了遵循以上格式规范外，还应严格按照以下风格、结构和逻辑撰写，全程保持幽默风趣、接地气，贴合程序员视角，拒绝生硬说教，同时保证干货满满、可落地：

### 风格要求（核心重点）
1.  语气：幽默接地气，多用程序员日常吐槽（比如"屎山代码""挨骂""加班""埋坑"），口语化但不随意，专业但不晦涩，短句为主，避免长句和废话；
2.  视角：站在程序员角度，结合日常写代码的痛点（比如3个月看不懂自己写的代码、被同事吐槽烂代码、改bug牵一发而动全身），引发共鸣；
3.  细节：多用"救命技能""看完想打人""清爽到哭"等接地气表述，穿插轻松调侃，不抒情、不鸡汤，每段都有信息增量。

### 结构参考（非固定框架，可灵活增减）
以下为核心结构参考，可根据书籍类型、内容多少灵活调整，允许适当增加或删减模块，但需保证核心逻辑连贯、干货完整：
1.  开头3要素（建议包含，可灵活简化）：
    - 核心一句话总结：用幽默的方式提炼书籍核心价值，点出"解决什么痛点"；
    - 适合人群：明确区分适合/不适合的人，贴合书籍定位（比如程序员相关书籍就分小白、中级、资深开发者）；
    - 最大收获：直白说明读完能得到什么实际好处（比如少加班、少背锅、提升职场竞争力）。

2.  开篇引入（建议包含，可灵活调整篇幅）：
    - 用程序员日常崩溃场景切入（比如自己写的代码看不懂、被烂代码折磨），自然引出书籍作者和书籍核心立意，可适当引用作者观点并加以调侃。

3.  核心干货（重中之重，建议保留，可灵活分点）：
    - 分点拆解书籍核心技巧（每点一个清晰小标题，带幽默表述，比如"命名——别瞎起名字，不然同事会骂你‘没文化’"）；
    - 每个技巧可包含：核心原则+反面例子（标注❌，配简单吐槽）+ 正面例子（标注✅，说明优势）+ 重点提炼（分点，简洁好记）+ 实践指南（可落地，结合日常开发场景）；（可根据技巧复杂度灵活增减例子或提炼内容）
    - 例子优先用程序员熟悉的代码案例（比如Java代码），其他类型书籍可替换为对应领域案例，确保真实可参考。

4.  全书框架总结（建议包含，可灵活简化）：
    - 极简版，分点提炼书籍核心逻辑，不冗余，方便复习（可根据书籍复杂度调整为3-5点）。

5.  个人思考（建议包含，可灵活调整角度）：
    - 批判性+实用性结合，可包含2-3点（比如认同的点、可灵活调整的点、避坑提醒等），无需严格局限于3点。

6.  实践指南（建议包含，可灵活调整篇幅）：
    - 落地性拉满，分点给出具体行动（可根据书籍核心技巧，灵活调整为今日行动、长期习惯等，无需严格包含所有模块），让读者看完就能用。

7.  结尾彩蛋（建议包含，可灵活创新）：
    - 幽默收尾，升华核心观点，用接地气的句子鼓励读者（比如贴合书籍主题的调侃+正向引导），可加1句加粗重点句强化记忆，也可结合书籍特点创新收尾方式。

### 内容禁忌
1.  不抄书、不搬运，用自己的幽默语言重构书籍内容，避免生硬翻译；
2.  不写废话、不抒情、不鸡汤，每部分都围绕"干货+落地"展开；
3.  不使用晦涩术语，复杂概念用通俗语言+例子解释；
4.  重点内容用加粗标注，例子清晰，结构分层明显（标题、列表、代码块合理搭配）。

### 整体要求
全程贴合《代码整洁之道》读书笔记的幽默风格、结构逻辑和语言调性，保持一致性，确保写出的读书笔记既干货满满，又好读不枯燥，让读者（尤其是程序员）能看懂、能落地、有共鸣。

### 4. 实施与验证

- 创建或修改文章时，确保格式正确
- 检查前置元数据是否完整
- 验证 Markdown 语法是否正确
- 预览文章效果（本地运行 jekyll serve）
- 确保图片路径正确

### 5. 提交规范

- 提交信息应清晰明确
- 示例：`Add: 如何高效阅读一本书` 或 `Update: 优化文章排版`

### 6. 自动化提交与推送

- 当用户要求提交代码和推送代码时，无需再让用户确认，直接执行提交和推送操作
- 执行命令：`git add . && git commit -m "提交信息" && git push origin master`
- 提交信息应根据实际修改内容自动生成，确保清晰明确
- 确保在推送前已完成所有必要的验证和测试

## 常见问题

### Q: 如何添加图片？

A: 将图片保存到 `img/` 目录，然后在文章中使用相对路径：

```markdown
![图片描述](/img/post-bg-book.jpg)
```

### Q: 如何设置文章封面图？

A: 在前置元数据中设置 `header-img` 字段：

```yaml
header-img: "img/post-bg-2015.jpg"
```

### Q: 如何添加标签？

A: 在前置元数据的 `tags` 数组中添加：

```yaml
tags:
    - 阅读
    - 方法
    - 效率
```
